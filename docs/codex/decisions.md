# Accepted Decisions

## D-001 Source Repository

Accepted: `/Users/suzukimakoto/plugins/coding-agents` is the source repository candidate for the home-local `coding-agents` plugin.

Impact: edit this source repo, not `~/.codex/plugins/cache/`, for plugin source changes.

## D-002 CWD Is Jobsite

Accepted: the plugin treats the current project cwd as the jobsite unless the user explicitly names another root.

Impact: every run starts by resolving and inspecting the jobsite before planning implementation.

## D-003 Project Intake First

Accepted: the first workflow phase is project intake.

Impact: Codex must read applicable instructions, inspect repo shape, check Git state, and identify existing `docs/codex` files before edits.

## D-004 Empty Specialist Warm Pool

Accepted: plugin activation uses an empty specialist warm pool design.

Impact: specialist contexts begin without durable task state and receive scoped assignments before work.

## D-005 Subagent Isolation Keys

Accepted: subagent work is isolated by `task_id`, `epoch`, and `scope`.

Impact: task identity, restart boundary, and allowed work boundary must be explicit in each assignment.

## D-006 Restart Or Retire By Default

Accepted: subagent context reuse is exceptional; restart or retire is the default after meaningful boundaries.

Impact: stale context and scope drift are handled by reset, not silent continuation.

## D-007 Docs/Codex Workflow SSOT

Accepted: `docs/codex/task.md`, `todo.md`, `decisions.md`, and `audit.md` are read and updated during the workflow.

Impact: accepted decisions become specification, TODO drives execution, and audit records verification state.

## D-008 Parent And Subagent Roles

Accepted: the parent owns decisions and integration; subagents provide research, implementation, and verification material.

Impact: subagents do not produce final policy or final user-facing synthesis.

## D-009 Marketplace Deferred

Accepted: `~/.agents/plugins/marketplace.json` is not edited during initial source setup.

Impact: marketplace registration remains a TODO after the initial commit.

## D-010 Runnable MVP Before Cache

Accepted: the first runnable Coding Agents implementation is a source CLI, not marketplace/cache activation.

Impact: use `node /Users/suzukimakoto/plugins/coding-agents/bin/coding-agents.mjs` directly until plugin activation is explicitly requested.

## D-011 Target Docs Generation

Accepted: `intake` creates or updates target project `docs/codex/README.md`, `project.md`, `task.md`, `todo.md`, `decisions.md`, `audit.md`, `assignments.md`, and `handoff.md`.

Impact: the MVP produces usable workflow state without MCP servers, hooks, app connectors, or real subagent process orchestration.

## D-012 Generated Section Safety

Accepted: generated files and sections use the `Generated by coding-agents MVP` marker.

Impact: generated content can be safely replaced on rerun, while existing non-generated target files are preserved with a generated section appended or updated.

## D-013 GUI Plugin Registration

Accepted: `coding-agents` is registered in the local GUI plugin marketplace and refreshed into the Codex plugin cache from the source repository.

Impact: marketplace/cache activation is now part of the explicit user-approved workflow; source remains `/Users/suzukimakoto/plugins/coding-agents`, not the cache copy.

## D-014 Subagent Runtime Contract Location

Accepted: warm-pool lifecycle states, isolation key schema, restart/retire rules, and parent integration output format are documented in `docs/codex/assignments.md`.

Impact: `assignments.md` is not only a generated role list; it also carries the stable assignment contract that future runner or MCP implementation must satisfy.

## D-015 Subagent Reuse Is Explicitly Exceptional

Accepted: specialist context reuse requires unchanged `role`, `task_id`, `epoch`, and `scope`, plus a recorded parent reason.

Impact: implementation must prefer fresh assignment or retirement over silent continuation when context may be stale.

## D-016 Broad Runtime Support Is Backlog, Not Assumed Behavior

Accepted: hooks, MCP/app support, broad scheduler behavior, and richer runner orchestration are not present in the current MVP; real child Codex process spawning exists only through the explicit `--runner codex-cli` opt-in defined by D-019.

Impact: docs may describe future runtime contracts, but audit must not claim hooks, MCP/app integration, broad scheduling, or default real spawning exists until implemented and verified.

## D-017 CLI Runner Packet Log

Accepted: `docs/codex/runner.md` is the source CLI packet log for issued assignments, parent-integration packets, and deferred process-orchestration skeletons.

Impact: `assign`, `collect`, and `run` may append packet material there, while `assignments.md` remains the stable role and schema contract.

## D-018 Default Process Spawning Deferred

Accepted: when `--runner` is omitted, the current `run` / `orchestrate` command records a skeleton and reports `spawned: false`.

Impact: the CLI can prove assignment handoff shape by default without pretending hooks, MCP, app integration, broad scheduling, or implicit real subagent process orchestration exists; explicit real child Codex process spawning is covered separately by D-019.

## D-019 Opt-In Codex CLI Runner

Accepted: default `run` / `orchestrate` behavior remains the existing `spawned: false` skeleton, while `run --runner codex-cli` opts into a non-interactive local `codex exec` child process.

Impact: the runner boundary is explicit, uses Node.js standard libraries only, runs with cwd from `--cwd`, passes a strict scoped prompt, captures the child last message through a temporary output file, and records normalized success or failure in `docs/codex/runner.md` without claiming success on unavailable `codex`, timeout, or nonzero exit.

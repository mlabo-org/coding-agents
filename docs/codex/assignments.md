# MVP Role Assignment Contract

The runnable MVP must generate 14 warm-pool role assignments for each target task:

- Intake
- Repo Mapper
- Requirements
- Planner
- Architect
- Implementer
- Test Runner
- Reviewer
- Risk Guard
- Docs Keeper
- UX
- Dependency
- DevOps
- Auditor

Each generated role entry must include `role`, `status`, `task_id`, `epoch`, `scope`, `assignment`, and `expected_output`.

## Subagent Runtime Design

### Warm-Pool Lifecycle States

- `empty`: no specialist context exists yet; only the parent-held role template exists.
- `assigned`: the parent has issued `role`, `task_id`, `epoch`, `scope`, `assignment`, and `expected_output`.
- `active`: a specialist is working inside the assigned scope.
- `blocked`: the specialist cannot continue without parent clarification, missing access, or a failed prerequisite.
- `returned`: the specialist has produced parent-integration material and should not keep extending the task silently.
- `retired`: the specialist context is closed and must not be reused for new scope or stale premises.

### Isolation Key Schema

- `task_id`: stable user-visible work unit. It changes when the user-visible task changes.
- `epoch`: restart boundary. It changes when assumptions, accepted decisions, task framing, or verification state become stale.
- `scope`: explicit allowed files, tools, commands, or investigation surface. It must be narrow enough for review and audit.
- Each assignment must repeat all three keys. A missing key is an invalid assignment, even if the surrounding task appears obvious.

### Restart And Retire Rules

- Retire by default after meaningful task completion, scope change, failed verification, stale premise discovery, or handoff to another worker.
- Restart by issuing a fresh assignment with a new `epoch` when prior context could bias the next result.
- Reuse is allowed only when `role`, `task_id`, `epoch`, and `scope` are unchanged, the parent explicitly records why reuse is safe, and no verification or user decision has invalidated the context.
- A blocked specialist returns a concise blocker packet to the parent; it does not broaden its own scope.

### Parent Integration Output

Every specialist return should be concise and shaped for parent integration:

- `status`: completed, blocked, partial, or failed.
- `findings`: key facts or issues, ordered by importance.
- `changed_files` or `proposed_files`: exact paths, or `none`.
- `verification`: commands/checks run and skipped checks with reasons.
- `blockers`: decisions, access, missing inputs, or conflicting instructions.
- `assumptions`: unresolved assumptions that the parent must not report as facts.
- `next`: the smallest useful next action.
